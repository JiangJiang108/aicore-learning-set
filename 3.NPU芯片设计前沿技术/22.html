22、NPU安全与隐私：侧信道攻击与防御、模型窃取与保护、可信执行环境(TEE)在NPU中的应用、硬件木马检测

说实话，做NPU芯片设计这些年，我最怕听到的一句话就是：「芯片被攻破了」。

以前大家觉得AI芯片就是个算力工具，安全离得远。但现在不一样了——模型就是资产，数据就是命脉。你花几千万训出来的大模型，别人通过侧信道几分钟就能把参数偷走。这可不是危言耸听，我在一个客户项目里亲眼见过。

好，咱们今天就把NPU安全这件事掰开揉碎了讲。四个维度：侧信道、模型保护、TEE、硬件木马。每个都是硬骨头。

核心观点：NPU安全不是「加个锁」那么简单。它是在性能、面积、功耗和安全之间找平衡。你加一层防护，推理延迟可能就翻倍。怎么取舍？这才是真功夫。

22.1 侧信道攻击与防御

侧信道攻击，说白了就是「不直接撬锁，而是听锁芯转动的声音」。NPU在计算时，功耗、电磁辐射、执行时间都会泄露信息。

22.1.1 攻击原理

NPU做矩阵乘法时，不同数值的功耗曲线不一样。比如权重为0和权重为127，MAC单元的翻转率完全不同。攻击者采集几千条功耗轨迹，用差分分析就能反推出模型参数。

我遇到过最夸张的一次——一个团队用示波器+开源工具，半小时就把一个8位量化的MobileNet权重恢复出80%。嗯，当时甲方脸都绿了。

22.1.2 常见攻击类型

  • 功耗分析攻击：采集芯片运行时的电流变化。SPA（简单功耗分析）看单条曲线，DPA（差分功耗分析）做统计。
  • 电磁辐射攻击：用近场探头扫描芯片表面。NPU的MAC阵列是强辐射源，很容易定位。
  • 时序攻击：利用不同输入导致的计算时间差异。比如某些激活函数对特定值有快速路径。
  • 缓存侧信道：在共享内存系统中，通过缓存命中/未命中时间差推断邻居任务的数据。

22.1.3 防御手段

我个人习惯把防御分成三级：

级别 方法 代价 适用场景
L1 掩码 对中间结果加随机掩码，计算完再去除 面积+15%，功耗+20% 高安全需求
L2 隐藏 固定执行时间、插入伪操作 性能下降30-50% 中等安全
L3 噪声注入 在电源网络中加随机噪声发生器 面积+5%，效果有限 低成本方案

避坑指南：我曾经在一个项目里只做了L3噪声注入，结果被客户的安全审计一枪毙掉。后来才明白——噪声注入只能提高攻击成本，不能从根本上消除信息泄露。如果你做金融级AI芯片，至少要做到L1+L2组合。

22.2 模型窃取与保护

模型窃取，比你想的容易得多。攻击者不需要拿到你的模型文件，只需要通过API反复查询，就能训练出一个「影子模型」，精度能达到原模型的95%以上。

22.2.1 模型窃取手法

  • 查询攻击：构造大量输入，收集输出logits，用这些数据训练替代模型。
  • 侧信道辅助窃取：结合功耗/电磁信息，加速参数提取。我在实验室做过实验，侧信道辅助下只需要1/10的查询量就能完成窃取。
  • 内存转储：如果攻击者能拿到系统级权限，直接从DDR里dump模型权重。NPU的权重通常以明文形式存在。

22.2.2 模型保护技术

这里我重点讲三个实用方向：

1. 模型加密与解密引擎

模型存在外部存储时必须是密文。NPU内部集成解密引擎，只在计算前解密一行权重。我参与的一个芯片里，我们用了AES-256-GCM，每次只解密一个tile（比如64x64），计算完立即擦除。

// 伪代码：NPU解密流程
void load_weight_tile(uint8_t *ciphertext, uint32_t tile_id) {
    uint8_t key[32] = {0x...}; // 硬件绑定的密钥
    uint8_t iv[12] = {0x...};  // 每次不同
    
    // 解密一个tile
    aes256_gcm_decrypt(ciphertext, weight_buf, key, iv);
    
    // 启动MAC计算
    npu_compute(weight_buf, tile_id);
    
    // 计算完成后立即清零
    memset(weight_buf, 0, TILE_SIZE);
}

2. 模型水印与指纹

在模型参数中嵌入不可见的水印。如果模型被泄露，可以通过水印追踪来源。我见过一种做法——在BatchNorm的gamma参数里嵌入二进制序列，对精度影响小于0.1%。

3. 查询限流与扰动

对API查询做频率限制，并在返回的logits中加入微小噪声。这样攻击者收集到的数据「不准」，训练出来的影子模型精度会大幅下降。

注意：模型水印不是万能的。攻击者可以通过微调、剪枝、量化等手段「洗掉」水印。所以水印只能作为事后追溯手段，不能替代加密和访问控制。

22.3 可信执行环境(TEE)在NPU中的应用

TEE这个概念在CPU上已经很成熟了——ARM的TrustZone、Intel的SGX。但在NPU上，事情变得复杂。为什么？因为NPU要访问大量数据，而且计算模式是流式的，不能像CPU那样轻易隔离。

22.3.1 NPU TEE架构

我推荐一种「双域隔离」架构：

  • 安全域：运行敏感模型和数据。NPU内部有独立的TLB和MMU，只能访问安全内存区域。
  • 非安全域：运行普通应用。无法访问安全域的内存和寄存器。

切换时，需要做完整的上下文保存和恢复——包括权重缓存、激活缓存、中间结果。嗯，这个开销不小，我实测大约需要5000个时钟周期。

22.3.2 关键设计点

组件 安全要求 实现方式
DMA控制器 只能从安全内存读取 增加安全位标记，DMA检查地址范围
权重缓存 非安全域不可见 物理隔离，独立电源域
中断控制器 防止安全域中断被劫持 安全中断专用向量表
调试接口 安全模式下禁用JTAG 硬件熔丝控制

我的经验：做NPU TEE最难的不是隔离本身，而是「安全切换的性能开销」。我曾经在一个项目里，因为频繁切换安全/非安全域，导致推理帧率从60fps掉到15fps。后来我们做了「批量切换」——把一批推理任务全部在安全域做完，再切回来。性能损失降到了5%以内。

22.4 硬件木马检测

硬件木马，是芯片设计中最隐蔽的威胁。它可能藏在RTL代码里，也可能在物理版图中。平时不激活，一旦触发就窃取密钥或破坏计算。

22.4.1 木马类型

  • 逻辑木马：在状态机或控制逻辑中插入后门。比如某个特定输入序列会触发「特权模式」。
  • 模拟木马：利用模拟电路特性。比如在电源网络上加一个电容，通过功耗变化泄露信息。
  • 物理木马：在版图中添加额外的金属层或晶体管。这种最难检测，需要X光或扫描电子显微镜。

22.4.2 检测方法

我把它分成「设计时检测」和「运行时检测」两类：

设计时检测：

  • 形式化验证：用数学方法证明设计中没有未定义的状态或路径。工具如Cadence JasperGold。
  • 代码审查：人工检查RTL代码。我团队有个老工程师，看一眼代码就能嗅出异常——「这个case语句怎么多了一个分支？」
  • 冗余分析：检查设计中是否有「死逻辑」——永远不被触发的电路。木马通常藏在这里。

运行时检测：

  • 功耗指纹：建立芯片正常运行的功耗基线。如果某次推理的功耗曲线偏离基线超过阈值，触发告警。
  • 时序监测：在关键路径上插入时序传感器。木马激活时通常会导致路径延迟变化。
  • 看门狗：监控NPU的状态机是否进入非法状态。一旦发现，立即复位并记录日志。

残酷的现实：没有任何一种方法能100%检测出硬件木马。尤其是针对先进工艺（7nm以下），木马可以做得极小，藏在金属层之间。所以工业界的做法是「多层防御」——设计时做形式化验证，流片后做物理检测，运行时做行为监测。三层叠加，把风险降到可接受水平。
