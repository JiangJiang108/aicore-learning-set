15、NPU验证与测试：功能验证方法、性能仿真与建模、硬件加速器测试策略、硅后调试与修复

NPU流片回来，不是终点，是起点。我见过太多团队，设计时意气风发，测试时愁眉苦脸。说白了，验证和测试才是真正见真章的地方。今天咱们就聊聊这块硬骨头怎么啃。

15.1 功能验证方法：从RTL到系统级

功能验证，核心就一句话：你的设计到底对不对？ 我习惯把验证分成三个层次，每个层次都有不同的玩法。

15.1.1 单元级验证

这是最基础的。每个MAC单元、每个数据通路、每个控制逻辑，都得单独测。我个人习惯用UVM（Universal Verification Methodology）搭环境，虽然学习曲线陡了点，但复用性是真的好。

核心验证点：

  • 计算精度：浮点/定点运算误差是否在容忍范围内
  • 数据流：输入输出是否对齐，有没有丢数据
  • 控制信号：握手协议、使能信号、复位行为是否正常

我曾经遇到过一个坑：某个MAC单元的乘法器在特定输入组合下会多出一个时钟周期的延迟。仿真跑了几百万个随机用例都没发现，最后是定向测试抓出来的。嗯，随机测试不是万能的。

15.1.2 子系统级验证

把几个单元拼起来，看它们能不能好好配合。比如卷积计算单元和激活函数单元之间的数据传递，这里最容易出问题的是背压（Backpressure）。

我的经验： 子系统验证一定要加延迟注入。模拟真实场景下，上游单元偶尔会卡住的情况。很多设计在理想延迟下跑得飞起，一加延迟就崩了。

15.1.3 系统级验证

把整个NPU子系统、缓存、DMA、甚至CPU都连起来。这时候验证的不只是功能，还有一致性。比如CPU写了一个权重到DDR，NPU读到的对不对？

我建议用黄金参考模型。跑一个算子，NPU的结果和软件模拟的结果逐比特对比。别嫌麻烦，这一步能省掉你后面80%的调试时间。

15.2 性能仿真与建模：提前看到瓶颈

功能对了，性能够不够？这个问题不能等流片回来再回答。我们需要在RTL阶段就建立性能模型。

15.2.1 周期精确模型

说白了，就是用软件模拟NPU每个时钟周期的行为。我常用的工具是SystemC TLM（Transaction Level Modeling）。

// 一个简化的性能模型示例
SC_MODULE(NPU_PerfModel) {
    sc_in_clk clk;
    sc_in<bool> start;
    sc_out<bool> done;
    
    // 内部状态
    int cycle_count;
    int total_mac_ops;
    
    void process() {
        while(true) {
            wait(); // 等待时钟
            if(start.read()) {
                cycle_count = 0;
                // 模拟计算延迟
                while(cycle_count < total_mac_ops / 64) {
                    wait();
                    cycle_count++;
                }
                done.write(true);
            }
        }
    }
};

这个模型虽然简单，但能帮你快速估算吞吐量和延迟。我在项目中用它来评估不同数据复用策略对性能的影响，比跑完整RTL仿真快了两个数量级。

15.2.2 性能瓶颈分析

建好模型后，就要找瓶颈了。常见的NPU性能瓶颈有：

瓶颈类型 表现 常见原因
计算瓶颈 MAC利用率低 数据供应不足，或者算子粒度太小
存储瓶颈 频繁的缓存未命中 数据局部性差，或者缓存容量不够
带宽瓶颈 DMA长时间占用总线 外部DDR带宽不够，或者数据搬运策略不合理

注意： 性能模型再准，也只是模型。我见过一个项目，模型预测利用率90%，实际芯片只有60%。原因是模型没考虑控制逻辑的额外开销。所以，模型要不断用RTL仿真结果校准。

15.3 硬件加速器测试策略：从仿真到FPGA

仿真跑得再好，也不如真刀真枪干一场。FPGA原型验证是NPU测试中不可或缺的一环。

15.3.1 仿真测试

这是最传统的。跑定向用例、随机用例、边界用例。我习惯把测试用例分成三类：

  • 冒烟测试： 最基本的，比如复位、基本运算。不过就停。
  • 功能测试： 覆盖所有指令和操作模式。
  • 压力测试： 最大数据量、最复杂计算图、最恶劣的时序。

我曾经在压力测试中抓到一个bug：当连续处理1000个3x3卷积时，内部状态机在某个特定条件下会跳错。这个bug在功能测试中完全没暴露，因为功能测试只跑几个卷积就停了。

15.3.2 FPGA原型验证

FPGA跑得比仿真快得多。一个复杂的神经网络，仿真可能要跑几个小时，FPGA上几秒钟就出结果了。

但FPGA也有坑：

  • 资源有限： 大型NPU可能放不下，需要分模块验证
  • 时序不同： FPGA的布线延迟和ASIC不一样，有些时序问题在FPGA上不出现，在ASIC上出现
  • 调试困难： 内部信号观察不如仿真方便

我的建议： FPGA验证时，一定要保留调试接口。比如通过JTAG或UART把内部状态寄存器读出来。否则出了问题，你只能对着示波器干瞪眼。

15.4 硅后调试与修复：流片后的最后防线

芯片回来了，插上板子，上电... 嗯，有时候它不工作。这时候就需要硅后调试了。

15.4.1 常见的硅后问题

我总结了几类：

  • 功能错误： RTL仿真没覆盖到的场景
  • 时序问题： 某些路径在特定PVT（工艺、电压、温度）下不满足时序
  • 功耗问题： 实际功耗超出预期，导致电压下降或温度过高
  • 良率问题： 部分芯片有缺陷，需要筛选或修复

15.4.2 调试手段

硅后调试，工具很重要：

工具 用途 我的经验
逻辑分析仪 抓取外部接口信号 看DDR读写时序，判断数据是否正确
示波器 测量模拟信号 检查时钟抖动、电源纹波
扫描链 内部寄存器状态读出 设计时一定要留扫描链，否则硅后就是盲人摸象
内置自测试（BIST） 存储器测试 SRAM最容易出问题，BIST能快速定位坏块

15.4.3 修复策略

发现问题后，怎么修？

  • 软件补丁： 如果是控制逻辑的问题，看看能不能通过修改驱动或固件绕过去。比如某个指令序列会触发bug，那就避免生成这个序列。
  • ECO（Engineering Change Order）： 如果是金属层的问题，可以通过修改金属掩模来修复。成本高，但比重新流片便宜。
  • 熔丝/反熔丝： 有些芯片设计了冗余单元，可以通过熔丝来替换有缺陷的部分。

血的教训： 我曾经有一个项目，硅后发现了bug，想通过软件补丁修复。结果补丁打上去，性能下降了30%。客户不干了。所以，软件补丁一定要评估性能影响，不能为了功能牺牲性能。

验证与测试，说白了就是一场与bug的持久战。你永远不可能把所有bug都消灭，但你可以通过系统的方法，把风险降到最低。我做了这么多年芯片，最大的体会就是：验证不是一个人的事，是整个团队的事。设计工程师、验证工程师、测试工程师，大家要坐在一起，把问题想透，把方案做全。
