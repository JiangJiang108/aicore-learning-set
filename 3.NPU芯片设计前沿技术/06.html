量化技术：训练后量化(PTQ)与量化感知训练(QAT)

量化这个话题，我做了快十年了。说实话，刚入行时觉得量化就是把32位浮点数砍成8位整数，简单得很。直到第一次把量化后的模型部署到芯片上，推理结果完全不对——嗯，那才叫真正的「社会毒打」。

今天咱们聊聊量化的两个主流方案：PTQ和QAT，以及INT8/INT4这些精度选择。我会结合自己踩过的坑，把技术细节掰开揉碎了讲。

为什么非量化不可？

先问个问题：为什么NPU一定要做量化？

你想想看，一个典型的AI模型，权重和激活值都是FP32。FP32乘法器面积大、功耗高。在边缘设备上，这根本扛不住。量化后，数据从32位降到8位甚至4位，存储带宽直接砍到1/4或1/8。计算单元也能塞更多。

我参与过一个智能音箱项目，量化前模型跑在DSP上要200ms，量化到INT8后降到35ms。用户体验从「说句话等半天」变成「秒回」。这就是量化的价值。

训练后量化（PTQ）

PTQ，说白了就是模型训练完了，我再做量化。不需要重新训练，省时省力。

基本流程是这样的：

  1. 准备好校准数据集（几百到几千张图片/样本）
  2. 跑一遍推理，收集每层的激活值分布
  3. 根据分布计算缩放因子（scale）和零点（zero point）
  4. 把权重和激活值量化到INT8/INT4

这里有个关键点：校准方法的选择。我见过太多人直接用min-max方法，结果遇到离群点，量化精度崩得一塌糊涂。

常用校准方法对比：

方法 原理 适用场景 我的经验
Min-Max 取最小最大值映射 分布均匀的数据 容易受离群点影响，慎用
KL散度 最小化量化前后分布差异 大多数CV模型 NVIDIA TensorRT默认方法，靠谱
百分位法 去掉极端值后映射 有离群点的场景 我习惯用99.9%百分位
MSE最小化 最小化量化误差的均方差 对精度敏感的任务 效果好但计算量大

代码示例，PTQ的简单实现：

import numpy as np

def quantize_ptq(tensor, num_bits=8, method='kl'):
    """训练后量化，以激活值为例"""
    # 收集激活值分布
    min_val = np.min(tensor)
    max_val = np.max(tensor)
    
    if method == 'min_max':
        # 直接映射
        scale = (max_val - min_val) / (2**num_bits - 1)
        zero_point = -min_val / scale
        
    elif method == 'percentile':
        # 去掉0.1%的极端值
        min_val = np.percentile(tensor, 0.1)
        max_val = np.percentile(tensor, 99.9)
        scale = (max_val - min_val) / (2**num_bits - 1)
        zero_point = -min_val / scale
    
    # 量化
    quantized = np.round(tensor / scale + zero_point)
    quantized = np.clip(quantized, 0, 2**num_bits - 1)
    
    return quantized.astype(np.uint8), scale, zero_point

避坑指南：我曾经在一个语义分割模型上直接用min-max量化，结果mIoU从0.72掉到0.31。换成KL散度后，恢复到0.68。校准方法的选择，直接影响PTQ成败。

量化感知训练（QAT）

PTQ虽然快，但精度损失有时扛不住。这时候就得请出QAT了。

QAT的核心思想：在训练过程中模拟量化效果，让模型自己去适应量化误差。说白了，就是让模型「提前知道」自己将来要被量化，提前调整权重。

具体做法：

  • 在前向传播时插入伪量化节点（fake quantize）
  • 把浮点权重先量化再反量化，模拟量化误差
  • 反向传播时，用直通估计器（STE）处理量化函数的梯度

这里有个细节：STE的梯度近似。量化函数不可导，STE的做法是让梯度直接通过，相当于把量化器当成恒等函数。听起来有点粗暴，但实际效果很好。

import torch
import torch.nn as nn

class FakeQuantize(nn.Module):
    """伪量化层，用于QAT训练"""
    def __init__(self, num_bits=8):
        super().__init__()
        self.num_bits = num_bits
        self.scale = nn.Parameter(torch.ones(1))
        self.zero_point = nn.Parameter(torch.zeros(1))
        
    def forward(self, x):
        # 量化
        q = torch.round(x / self.scale + self.zero_point)
        q = torch.clamp(q, 0, 2**self.num_bits - 1)
        # 反量化（模拟量化误差）
        x_q = (q - self.zero_point) * self.scale
        # STE：梯度直接通过
        return x + (x_q - x).detach()

注意：QAT训练时，学习率要调小。我习惯用原始训练学习率的1/10。学习率太大，量化参数震荡，模型很难收敛。

INT8 vs INT4 vs 混合精度

精度选择是个权衡题。INT8精度损失小，但压缩比只有4倍。INT4压缩比8倍，但精度损失大。

我的建议：

  • INT8：大多数场景的首选。精度损失通常在1%以内，硬件支持最成熟
  • INT4：适合对带宽极度敏感的场景，比如手机端语音模型。但要做好精度补偿
  • 混合精度：敏感层用INT8，不敏感层用INT4。我在一个检测模型上试过，精度只掉0.5%，但模型大小减少40%

混合精度怎么分？我分享一个经验：看激活值的分布。分布宽、方差大的层，用INT8。分布集中、接近零的层，用INT4。

量化误差分析与补偿

量化误差主要来自三方面：

  1. 截断误差：超出量化范围的值被截断
  2. 舍入误差：连续值映射到离散值时的精度损失
  3. 梯度误差：QAT中STE的近似误差

怎么补偿？我常用的方法：

  • 逐通道量化：每个卷积核单独算scale，比逐层量化精度高
  • 偏置校正：量化后统计输出偏差，在推理时加一个偏置项
  • 知识蒸馏：用FP32教师模型指导量化后的学生模型训练

一个真实案例：我在做人脸识别模型时，INT8量化后准确率从99.2%掉到97.8%。用了逐通道量化+偏置校正后，恢复到98.9%。虽然没完全恢复，但已经满足产品要求了。

硬件友好的量化方案

做量化不能只盯着算法精度，还得考虑硬件能不能跑。我见过太多算法工程师设计的量化方案，硬件团队看了直摇头。

硬件友好的原则：

  • 对称量化：zero point固定为0，省掉减法器。INT8对称量化在NPU上能快15%
  • 2的幂次scale：scale用2的幂次，乘法变移位操作。省面积省功耗
  • 统一量化参数：同一层内尽量共用scale，减少参数存储
  • 避免跨层重量化：相邻层用相同量化参数，省掉重量化单元

我记得有个项目，算法团队用了非对称量化+精细的逐元素scale。硬件实现时，每个乘法器前面都要加一个减法器，面积直接翻倍。后来改成对称量化+2的幂次scale，面积降了40%，精度只掉了0.2%。

量化技术，说到底就是在精度、速度和硬件成本之间找平衡。没有银弹，每个项目都要根据实际需求做取舍。

我个人习惯是：先用PTQ快速评估精度损失。如果损失在可接受范围内，直接上PTQ，省时省力。如果损失太大，再上QAT。至于精度选择，从INT8开始，不够再考虑混合精度。
