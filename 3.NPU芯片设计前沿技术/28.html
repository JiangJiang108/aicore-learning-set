28、NPU发展趋势：3D堆叠与存算一体融合、稀疏与量化协同优化、大模型专用NPU、开源硬件与开放指令集

各位同学，今天我们聊点“未来感”的东西。NPU发展到今天，单纯堆算力已经走不远了。我这些年做芯片，最大的感受就是：瓶颈不在计算，而在搬运。数据搬来搬去，功耗全耗在路上了。所以，接下来的几个趋势，说白了都是在解决同一个问题——怎么让数据离计算更近，怎么让计算更高效。

核心趋势一览：

  • 3D堆叠与存算一体融合 —— 打破“存储墙”
  • 稀疏与量化协同优化 —— 榨干模型冗余
  • 大模型专用NPU —— 为Transformer而生
  • 开源硬件与开放指令集 —— 生态的力量

1. 3D堆叠与存算一体融合：把存储“叠”到计算上

先说说3D堆叠。传统芯片是平面铺开的，CPU、NPU、DDR各占一块地方，数据通过PCB走线传来传去。我做过一个项目，算力明明够，但跑大模型时延迟就是降不下来。一分析，70%的时间花在等数据从DDR搬过来。这就是经典的“存储墙”。

3D堆叠的思路很简单——把存储芯片直接叠在NPU上面，中间用硅通孔（TSV）连接。距离从厘米级降到微米级，带宽翻几十倍，功耗却降一大截。

但光堆叠还不够，还得“存算一体”。什么意思？就是存储单元本身也能做计算。比如忆阻器阵列，存的是权重，算的是矩阵乘法。数据根本不用搬，在存储单元里就完成了乘累加。

我的经验：3D堆叠的散热是个大坑。我曾经有一版设计，NPU核心温度直接飙到110度。后来加了硅中介层和微流道散热，才压下来。做3D堆叠，热管理必须从一开始就规划，别等流片回来再想办法。

方案带宽功耗延迟成熟度
传统DDR~50 GB/s高~100 ns成熟
HBM（3D堆叠）~1 TB/s中~10 ns量产
存算一体（忆阻器）~10 TB/s低~1 ns实验阶段

2. 稀疏与量化协同优化：给模型“瘦身”

大模型动辄几百亿参数，全精度推理？不现实。所以稀疏化和量化成了标配。

稀疏化，说白了就是把权重里那些接近零的剪掉。我见过一个BERT模型，剪掉80%的权重，精度只掉了不到1%。但硬件要支持稀疏计算，得有专门的索引电路。你想想看，如果计算单元还在傻傻地乘零，那剪枝就白做了。

量化，就是把FP32变成INT8甚至INT4。我习惯在训练时就用“伪量化”模拟，这样部署时精度损失最小。但要注意，量化不是简单的截断，得做校准——找一批数据跑一遍，统计激活值的分布，再决定量化参数。

协同优化才是关键：稀疏和量化不是孤立的。稀疏后的模型，非零值分布更集中，量化误差更容易控制。反过来，量化后的模型，数值范围变小，稀疏剪枝的阈值更好选。我建议你们做优化时，把这两个步骤放在一个框架里迭代，效果比分开做要好得多。

# 伪代码：稀疏+量化协同优化流程
model = load_model()
for epoch in range(10):
    # 1. 稀疏剪枝
    mask = compute_sparsity_mask(model, threshold=0.01)
    apply_mask(model, mask)
    # 2. 量化校准
    calib_data = load_calibration_dataset()
    quant_params = calibrate_quantization(model, calib_data)
    # 3. 微调恢复精度
    model = finetune(model, quant_params)
    # 4. 评估
    acc = evaluate(model)
    print(f"Epoch {epoch}: accuracy = {acc:.2f}%")

3. 大模型专用NPU：为Transformer定制

通用NPU什么都能跑，但什么都不精。大模型火了之后，大家发现Transformer的计算模式很特殊——矩阵乘法占大头，但Attention里的Softmax和LayerNorm也很关键。

我参与过一个项目，专门为Transformer设计NPU。核心改动有三点：

  • 增加专用的Softmax硬件单元——用查找表+线性插值，比软件实现快10倍。
  • 优化矩阵乘法阵列——支持可变精度，比如在Attention里用FP16，在FFN里用INT8。
  • 片上SRAM要大——至少能放下一个层的权重和激活值，减少片外访问。

避坑指南：我曾经在设计中忽略了“KV Cache”的存储需求。大模型推理时，Key和Value要缓存下来，这个量很大。如果SRAM不够，就得频繁写DDR，延迟直接爆炸。所以，做Transformer NPU，KV Cache的带宽和容量一定要留够。

4. 开源硬件与开放指令集：生态的力量

最后聊聊生态。RISC-V这几年在AI领域越来越火。为什么？因为NPU的指令集太碎片化了，每家都有自己的私有指令。你想想看，换一家芯片，编译器、算子库全得重写，这谁受得了？

开源硬件的好处是：你可以基于RISC-V扩展自定义AI指令，比如矩阵乘指令、向量化指令。而且社区里已经有现成的工具链，不用从头造轮子。

我个人的习惯是：核心计算用自定义指令，控制逻辑用标准RISC-V。这样既保证了性能，又兼容了生态。比如我们团队做的NPU，矩阵乘法用自定义的vfmacc指令，而任务调度、中断处理全走标准RISC-V。

一点建议：如果你刚开始做NPU，别急着闭门造车。先看看开源项目，比如Gemmini（伯克利的开源NPU生成器）或者OpenASIP。站在巨人肩膀上，能少踩很多坑。
