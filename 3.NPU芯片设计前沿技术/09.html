9、多核NPU架构：同构多核与异构多核、任务调度与负载均衡、核间通信与同步、多核NPU的存储一致性

各位同学，今天我们来聊聊多核NPU架构。说实话，这个主题我琢磨了快十年。从最早的单核NPU，到后来做多核，踩过的坑真不少。你想想看，一个AI芯片里塞进几十个甚至上百个计算核，怎么让它们好好配合？这就是我们今天要解决的核心问题。

9.1 同构多核 vs 异构多核：选哪个？

先说说架构选型。我个人习惯把多核NPU分成两大类：同构和异构。

同构多核，说白了就是所有核长得一模一样。每个核都有相同的计算单元、缓存大小、指令集。我在做第一代多核NPU时就选了这条路。为什么？因为设计简单，软件好写。你写一个核的驱动，其他核直接复用。任务分配也省心，随便扔给哪个核都行。

但同构有个问题：资源浪费。比如有些任务需要大量矩阵乘法，有些任务需要大量激活函数计算。同构核只能平均分配资源，结果就是矩阵运算多的核忙死，激活函数多的核闲死。

异构多核就不一样了。每个核可以有不同的专长。比如我最近参与的一个项目，芯片里有三种核：

  • 大核：负责大矩阵运算，有超大SRAM和SIMD单元
  • 小核：负责激活函数和池化，功耗低
  • 专用核：专门做量化、反量化、数据格式转换

这样设计的好处很明显：每类任务都能找到最合适的核。但代价是什么？软件复杂度飙升。你得为每种核写不同的编译器、不同的调度策略。

我的建议：如果芯片规模在8核以内，同构就够了。超过16核，强烈建议考虑异构。我在一个32核的项目中硬用了同构，结果功耗比预期高了40%。

9.2 任务调度与负载均衡：别让核闲着

好，架构选好了，接下来就是怎么把任务分下去。这里有个关键问题：负载均衡。

我曾经遇到过一个真实案例：一个8核NPU跑ResNet-50，结果7个核在忙，1个核在发呆。为什么？因为任务分配太粗糙了，把整个卷积层扔给一个核，结果那个核计算量大，其他核只能干等。

解决思路其实就两个：

  • 静态调度：编译时就确定好每个核做什么。适合计算模式固定的场景，比如推理。
  • 动态调度：运行时根据核的负载情况动态分配。适合训练场景，因为计算图会变。

我个人更推荐混合调度。举个例子：

// 伪代码：混合调度策略
if (task_type == CONV_LARGE) {
    // 静态调度：直接分配给大核
    assign_to_big_core(task);
} else if (task_type == ACTIVATION) {
    // 动态调度：找当前最闲的小核
    core_id = find_idle_small_core();
    assign_to_core(task, core_id);
} else {
    // 负载均衡：按核的当前队列长度分配
    core_id = find_least_loaded_core();
    assign_to_core(task, core_id);
}

避坑指南：我曾经在动态调度中用了全局锁，结果性能反而下降了30%。后来改成无锁队列，每个核有自己的任务队列，性能才上去。记住：调度器本身不能成为瓶颈。

9.3 核间通信与同步：别让数据等太久

多核NPU最头疼的问题之一就是核间通信。你想想看，一个卷积层的数据可能分散在8个核上，下一层需要把这些数据拼起来。怎么传？

常见的通信方式有三种：

方式 延迟 带宽 适用场景
共享内存 低 高 小数据量、频繁通信
DMA 中 极高 大数据块传输
NoC（片上网络） 高 中 多对多通信

我在一个项目中用过共享内存做核间通信。当时觉得简单，结果发现缓存一致性问题让人崩溃。核A写了一个数据，核B读到的却是旧值。为什么？因为缓存没刷新。

解决方案是加同步屏障。比如：

// 核间同步示例
// 所有核执行到这里时，必须等所有核都到达才能继续
barrier_wait(&global_barrier);

// 然后才能安全地读取其他核的数据
data = shared_memory[neighbor_core_id];

注意：同步屏障不能滥用。我见过一个团队在每个操作后都加屏障，结果性能还不如单核。正确的做法是：只在数据依赖点加同步，其他时候让核自由运行。

9.4 多核NPU的存储一致性

最后聊聊存储一致性。这是个老生常谈的问题，但在NPU里尤其棘手。

为什么？因为NPU的计算模式跟CPU不一样。CPU是乱序执行，NPU是数据流驱动。数据流驱动意味着：数据到了，计算就开始了。如果数据不一致，计算就全错了。

我总结了几条经验：

  • 局部性优先：尽量让一个核处理连续的数据块，减少跨核访问。比如卷积的输入特征图，按通道切分给不同核。
  • 写回策略：NPU的缓存建议用写回（write-back）而不是写通（write-through）。写通虽然一致性容易保证，但带宽消耗太大。
  • 硬件一致性协议：如果芯片规模大（16核以上），建议用MESI或MOESI协议。虽然硬件复杂，但软件省心。

我记得有一次调试一个8核NPU，发现推理结果总是不对。查了三天，最后发现是两个核同时写同一个缓存行，导致数据覆盖。解决方案很简单：给每个核分配独立的缓存行，或者用原子操作。

核心原则：多核NPU的存储一致性，不是要保证所有时刻都一致，而是要保证在需要一致的时候一致。说白了，就是按需同步，别过度设计。

好了，多核NPU架构的内容就讲到这里。记住：架构选型、任务调度、核间通信、存储一致性，这四个点环环相扣。一个地方没做好，整个芯片的性能就上不去。
