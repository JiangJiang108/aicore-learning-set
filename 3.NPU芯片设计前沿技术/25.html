第25章：NPU设计自动化：硬件生成器与Chisel、高层次综合(HLS)在NPU中的应用、自动架构搜索(NAS)、设计空间探索

做NPU芯片设计这些年，我越来越觉得一个道理：设计自动化不是锦上添花，而是生存刚需。你想想看，一个现代NPU动辄几十个PE阵列、复杂的存储层次、灵活的数据流调度，如果全靠手写Verilog，那开发周期得拖到猴年马月去。

这一章，我就把NPU设计自动化的几个核心手段掰开揉碎了讲。包括硬件生成器（特别是Chisel）、高层次综合（HLS）、自动架构搜索（NAS），以及设计空间探索。这些工具和方法，说白了就是帮我们从“画电路”升级到“描述电路”。

25.1 硬件生成器：从手绘到代码生成

先聊聊硬件生成器。我个人习惯把传统RTL设计比作“手绘电路图”，而硬件生成器就是“用代码画电路图”。Chisel（Constructing Hardware in a Scala Embedded Language）是这里面最典型的代表。

Chisel不是一个新的硬件描述语言，它本质上是嵌入在Scala里的一套硬件构造库。你写的是Scala代码，但生成的是Verilog。这有什么好处？我举个例子：

// 一个可参数化的MAC阵列生成器
class MACArray(val rows: Int, val cols: Int) extends Module {
  val io = IO(new Bundle {
    val a = Input(Vec(rows, UInt(8.W)))
    val b = Input(Vec(cols, UInt(8.W)))
    val out = Output(Vec(rows, Vec(cols, UInt(16.W))))
  })
  
  // 生成二维MAC阵列
  val macs = Seq.tabulate(rows, cols) { (i, j) =>
    val mac = Module(new MAC())
    mac.io.a := io.a(i)
    mac.io.b := io.b(j)
    mac
  }
  
  // 连接输出
  for (i <- 0 until rows; j <- 0 until cols) {
    io.out(i)(j) := macs(i)(j).io.out
  }
}

你看，用Chisel写硬件，本质上是在写生成硬件的程序。我在项目中遇到过需要快速迭代不同尺寸PE阵列的场景——手写Verilog改参数要改半天，用Chisel改两个数字就行。

我的经验：Chisel的学习曲线确实有点陡，特别是Scala的函数式编程概念。但我建议你咬咬牙啃下来——一旦你习惯了用生成器思维设计硬件，就再也回不去了。我团队现在所有NPU的PE阵列、数据分发网络都是用Chisel生成的。

25.2 高层次综合：用C/C++描述硬件

再说HLS。高层次综合允许你用C/C++或SystemC来描述功能，然后自动转换成RTL。说白了，就是让软件工程师也能写硬件。

但这里有个坑——不是所有C代码都能综合成好硬件。我曾经见过一个团队把纯软件思维写的C代码扔进HLS工具，结果生成的电路面积大得离谱，时序也跑不通。

HLS在NPU设计中最适合的场景是：

  • 控制逻辑：状态机、调度器这类逻辑密集但数据通路简单的模块
  • 预处理/后处理：图像归一化、数据量化等计算量不大但逻辑复杂的部分
  • 快速原型验证：先HLS跑通功能，再手动优化关键路径

举个例子，一个简单的量化函数用HLS写：

// HLS实现的量化函数
void quantize(int8_t* out, float* in, float scale, int size) {
#pragma HLS INTERFACE m_axi port=out depth=1024
#pragma HLS INTERFACE m_axi port=in depth=1024
#pragma HLS PIPELINE II=1
  
  for (int i = 0; i < size; i++) {
#pragma HLS UNROLL factor=4
    out[i] = (int8_t)(in[i] * scale);
  }
}

嗯，这里要注意：HLS的pragma（编译指示）才是真正的精髓。PIPELINE、UNROLL、INTERFACE这些指令决定了最终硬件长什么样。我建议你花时间吃透这些pragma，而不是只关注C代码本身。

避坑指南：我曾经在一个项目里用HLS生成NPU的卷积计算核心，结果发现HLS对不规则数据流（比如稀疏卷积）的支持很差。生成的硬件要么面积爆炸，要么性能不如手写。所以我的建议是：计算密集且规则的部分用手写RTL或Chisel，控制逻辑和预处理用HLS。

25.3 自动架构搜索：让AI设计NPU

接下来聊个更前沿的——自动架构搜索（NAS）。这玩意儿最早火在神经网络架构搜索上，但现在已经被引入到硬件设计领域了。

NAS在NPU设计中的思路是这样的：

  1. 定义搜索空间：比如PE阵列大小、数据位宽、存储层次、数据流模式
  2. 选择搜索策略：强化学习、进化算法、贝叶斯优化等
  3. 性能评估：对每个候选架构进行快速评估（延迟、功耗、面积）
  4. 迭代优化：找到帕累托最优的架构配置

我参与过一个项目，用强化学习搜索NPU的片上存储分配方案。搜索空间包括：

搜索维度 取值范围 对性能的影响
PE阵列大小 4x4 ~ 32x32 计算吞吐量
输入缓冲区大小 2KB ~ 64KB 数据重用率
权重缓冲区大小 4KB ~ 128KB 权重加载频率
累加器位宽 16bit / 32bit 精度与面积
数据流模式 权重固定 / 输入固定 / 输出固定 数据移动开销

你想想看，如果手动遍历这个空间，每个配置都要综合、布局布线、跑仿真，那得几个月。但用NAS，我们可以用代理模型（比如线性回归或小神经网络）来快速预测性能，把评估时间从小时级降到秒级。

核心观点：NAS不是要取代设计师，而是帮设计师缩小搜索范围。我习惯的做法是：先用NAS跑一轮粗搜索，找到几个有潜力的区域，然后人工介入做精细调优。人机协作，效率最高。

25.4 设计空间探索：系统化的权衡分析

最后说说设计空间探索（DSE）。这跟NAS有点像，但更偏系统级和工程化。DSE的核心是回答一个问题：在功耗、性能、面积（PPA）的三角约束下，哪个架构配置最优？

我在做NPU设计时，常用的一种DSE方法是帕累托前沿分析。具体做法是：

  • 生成一批候选架构配置
  • 对每个配置评估延迟、功耗、面积
  • 找出那些“在某个指标上最优，且其他指标不差”的配置
  • 这些配置构成帕累托前沿

举个例子，假设我们在探索NPU的PE阵列大小和时钟频率：

# 伪代码：设计空间探索
configs = []
for pe_size in [4, 8, 16, 32]:
    for freq in [200, 400, 600, 800]:  # MHz
        perf = estimate_performance(pe_size, freq)
        power = estimate_power(pe_size, freq)
        area = estimate_area(pe_size, freq)
        configs.append((pe_size, freq, perf, power, area))

# 找出帕累托最优配置
pareto_front = find_pareto_front(configs, metrics=['perf', 'power', 'area'])

嗯，这里有个实际经验：DSE的精度取决于评估模型的精度。如果你用太粗糙的模型（比如线性估算），搜出来的结果可能跟实际差很远。我建议至少用RTL综合后的门级网表来做评估，虽然慢一点，但靠谱得多。

我的工作流：先用Chisel生成参数化的NPU模板，然后用Python脚本驱动DSE工具，自动修改参数、调用综合工具、收集结果。整个过程可以跑在服务器集群上，一晚上能探索几百个配置点。第二天早上看结果，挑出最好的几个做详细验证。

25.5 总结：自动化不是万能，但不用自动化是万万不能

讲了这么多，我想表达的核心观点其实很简单：NPU设计已经复杂到无法靠“手工作坊”模式来完成了。Chisel帮你快速生成硬件结构，HLS帮你加速控制逻辑开发，NAS帮你找到最优架构，DSE帮你系统化地做权衡。

但我也要泼点冷水——自动化工具不是银弹。我见过有人完全依赖HLS，结果生成的电路效率极低；也有人迷信NAS，搜出来的架构根本没法物理实现。工具是辅助，真正的判断力还是来自你对硬件本质的理解。

最后分享一个我个人的习惯：永远保留一个“手动挡”选项。自动化工具生成的结果，我一定会手工检查关键路径的时序、功耗热点、数据流瓶颈。毕竟，流片一次的成本够买好几台服务器了，多花点时间验证，值。

一句话总结：用Chisel生成结构，用HLS加速控制，用NAS搜索空间，用DSE做权衡——但最终拍板的，还是你脑子里的那个“硬件直觉”。
